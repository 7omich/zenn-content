## はじめに

こんにちは。株式会社Luup CTOの岡田[(@7omich)](https://x.com/7omich)です。
2025年ももうすぐ終わり、今年の Advent Calendar も無事に最後の記事となりました。

2025年は「AI エージェント元年」と呼ばれた激動の1年でしたが、Luup の社内においてもエージェントを始めとする各種AIツールの活用が大きく進み、AI ネイティブな事業開発組織への変革が始まった印象的な年でした。  
現実世界の市場にたくさんのマイクロモビリティの車両を配置してソフトウェア基盤から制御し、メンテナンスやバッテリー交換など多種多様な現場オペレーション業務を通じて日々管理している私たちの事業が、どのように AI エージェントを導入し活用しているのかご紹介したいと思います。

## 開発組織での活用

Luup ではプロダクト開発組織におけるAIによる業務活用がソフトウェアエンジニアを中心に先へ進み、その後各部署でのAI活用やそのための業務フロー改革が進んでいきました。  

### コーディング関連のサポート

Luup のソフトウェア開発でのAI活用は今年の5月頃から本腰を入れていきました。  
やや世の中的には出遅れた印象もありますが、「とりあえず積極的に触ってみる」という方針のもと予算を確保し、以下のツールを法人契約して各人が好きなものを自由に使える状態にしてきました。

- Devin
- Claude Code
- Cursor
- Cline
- GitHub Copilot
- Gemini

現在の Software 部の開発においては、Claude Code と Devin の併用が一番よく使われているようです。  
サーバーサイド、モバイルアプリ開発 (iOS, Android) いずれについても、直近の Top Commiter は Claude になっています。

下のキャプチャは iOS 開発リポジトリにおける Merge された PR の数の変化ですが、 **導入が進んだ5月頃から明らかに生産性が向上している** のがわかります。  
10月頃からは iOS 開発主務のメンバーが3人→2人へ減少したにも関わらず、同等の生産性を維持できています。（外れたメンバーはリードエンジニアでしたが、基盤側の整備や自動テストの拡張など、より開発者体験や品質向上の分野へコミットできるようになりました！）

![ios-merged-pr-count](/images/etc-okada-20251225/ios-merged-pr-count.png)

現在のところ AI エージェントによる業務改革の天井や、どれほどの投資対効果が得られるかなどの本質は業界的にもそこまでクリアに見えてない一方で、もたらす影響度の高さや、この機の変革を逃した場合の機会損失の大きさは明白です。  
ですので、とにかく今は AI ネイティブな開発体制を作って知見をいち早く蓄積していくことを優先し、生産性の測定やコスト最適化は走りながら同時並行で行っていくことにしています。  
後述する経営陣の理解もあり、現状少なくないコストを掛けて AI 活用を推し進めていますが、今はこのスタンスが正解かと考えています。

### コードレビューにおける活用

初期はエージェントがもたらす生産性爆増効果に沸き立っていた開発チーム内でも、日が経つにつれ少しずつ課題が見えてきました。  
開発規模が大きく関わる人数の多いリポジトリ、例えばバックエンドの開発においては PR が大量に生成されるものの中々 Merge まで至らず、結果的に世の中へデプロイできる機能・施策のペースが大きく変わっていない現実がありました。

![server-merged-pr-count](/images/etc-okada-20251225/server-merged-pr-count.png)

大きな課題の1つが、 **コードレビューの工数不足** です。  
そこで、GitHub Action 上で Claude Code による自動レビューのプロセスを挟み、標準的なレビューに加えて改善提案まで自動で提示されるようにしました。

Claude がレビュー時に読み込むルールファイルにはフロントエンド、バックエンドそれぞれのプロジェクトのアーキテクチャや開発原則、コーディング規約などが参照可能になっているため、簡単なコード品質や言語仕様に関するチェックだけでなく、現行のアーキテクチャへの適合性も検査されるようになっています。

加えて、開発にかかる機能要件や仕様書は GitHub Projects にて必ず Issue が切られて管理されている（リリースバージョンの管理やトラッキングもこのリポジトリで行われている）ので、当該 Issue を参照することで PR の内容が機能要件に漏れなく適合しているかどうかもレビューが行われます。

これらの工夫により、複雑かつ大規模な開発の中でも AI による生産性向上を享受できる体制を整えています。

また、もう1つの課題として **コンテキストウィンドウの拡大によるコード品質の低下** がありましたが、こちらは DDD によるコンテキストエンジニアリングや、モジュラーモノリスの採用によって解決を図っています。  
詳しくは以下の記事を参照ください。

https://zenn.dev/luup_developers/articles/server-yasumoto-20250909

### QA業務における活用

Luup では QA と SRE のチームを統合して、機能要件と非機能要件、リリース前の不具合検出とリリース後の継続的検証をそれぞれ高度に実現したソフトウェア品質の管理を目指しています。

https://zenn.dev/luup_developers/articles/sre-gr1m0h-20251125

その一環として、QA 業務においてもテスト設計書の作成や不具合調査の効率化にも AI が全面的に活用されています。  
前述の通り、機能要件がまとめられている Issue を介して Devin や Claude が仕様の情報を取りに行けるため、不具合の問い合わせがあった場合の仕様確認や原因究明に大きく役立っています。エンジニアバックグラウンドがあるわけではない QA メンバーでも、日々当たり前のように Devin を使い込んでいる環境になっています。

E2Eテストの自動生成などのチャレンジも進めており、開発速度が爆増した分の後工程を支えるための基盤整備に力を入れています。

### プロダクトマネジメント、デザイン領域での活用

技術者だけでなく、プロダクトマネージャーやデザイナーも AI 活用が進んでいます。  
プロダクトマネジメントにおいては、社内システムの顧客である各部門のメンバーから「解決したい課題」のヒアリングを行い、PRD に落とし込んでくれる Gem を作成し、要件の整理に役立てています。

https://zenn.dev/luup_developers/articles/server-watanabe-20251222

デザイナーも後述の全社 AI 基盤を活用して、現行のイラストシステムを簡単に拡張する仕組みを整えたり、Figma MCP を活用したデザイン→コード変換の自動化にチャレンジしたりしています。

## 全社での活用

といっても、実際に事業をつくって運営していくのは、勿論ソフトウェア開発関連の人員だけではありません。  
全社的に AI による生産性向上を実現し、顧客への価値提供を加速・最大化させてこそ、AI 活用が出来ていると呼べるものです。

### 個人レベル

Luup では社内の業務基盤および IdP として Google Workspace を全面的に採用しているため、Enterprise プランのサービスとして Gemini の最新モデルが全社員自由に使えるようになっています。  
加えて、Google Drive, Slack, Notion, … 等々に散らばる社内情報を一元的に統合して再利用できるよう、AI プラットフォーム「[Gemini Enterprise](https://cloud.google.com/gemini-enterprise?hl=ja)」も全社的に導入しました。

https://luup.sc/news/2025-11-26-gemini/

Web 検索や逐次のコンテキスト注入による一般論的な回答だけでなく、社内の資料や会議議事録、Slack 上での議論などを横断的に検索して導かれる精度の高い回答や生成コンテンツが得られます。  
これによってビジネスサイドや営業部署、現場オペレーション、コーポレート部門に至るまで全面的に AI/LLM による恩恵を受けることができるようになり、全社単位での生産性が大きく向上しました。

![gemini-effect-voice](/images/etc-okada-20251225/gemini-effect-voice.png)

Luup はハードウェア、営業、オペレーション、渉外などさまざまな業務ドメインが存在する複雑性の高い事業会社なので、こういった情報の一元化にはとても大きな価値があります。  
また、各分野のプロフェッショナル1人1人が大きな創造性を発揮できるようになるため、少数精鋭スタートアップという組織特性にも強くフィットするものだと考えています。

### 業務フローレベル

ただ、これで終わりではありません。  
上記はあくまで個人の AI 活用レベルを上げているだけの話です。AI ネイティブな事業開発を行える組織になるためには、こんなことではまだ足りません。

![ai-level](/images/etc-okada-20251225/ai-level.png)

上記は、Gemini Enterprise の正式導入を社内周知する全社定例で、私と CEO で作ったスライドの抜粋です。  
個人が AI/LLM を日々のタスクの効率化に使いこなせているのはあくまで土台。その先に **エージェントやワークフローを駆使しての業務フロー自体の効率化、果ては自動化を実現していくこと** が重要な本筋であり、さらに言えば事業そのものの価値にどう組み込まれていくかを考え実現していくことが最上位の命題だと考えています。

営業活動の複雑な業務体系とデータ基盤、サポート業務の膨大なトラフィックをさばく業務フロー、etc…  
これらを深く入り込んで理解し、業務構造とデータの流れを把握した上で AI/LLM を適用した最適な流れに組み替えることや、プロダクトのデータ基盤や ID 基盤と効果的に接続していくこと。  
運営にかかる実業務がさまざま存在する Luup の事業経営においては、ソフトウェア開発を大幅に加速させることよりもこちらの方がより大きなインパクトとなり得る機会がいくつも存在します。

現在は Software 部の AI に造詣の深いメンバーがこれらの各部門のチームに入り込み、定期的なディスカッションを通じて業務フローの再設計をしています。  
定量的にも、月に数百時間レベルでの業務効率化がいくつも進んでいる状態になっています。

### 事業レベル

最も重要なのは、 **事業そのものに AI がどう活かされるか** です。  
Luup はサービス提供の前提に AI や LLM による生成物が存在するプロダクトではないですが、モデル導入による最適化や、外部との MCP 接続などにより生み出せる価値は広く考えることができます。

もちろん具体的にここで書くことはできませんが、このレイヤーでの議論も経営陣による定期的な AI 戦略ミーティングの中で、前述の「個人レベル」「業務フローレベル」の進捗状況の確認と共に、国内外の AI プロダクトや業界動向のリサーチ結果も踏まえて行われています。  
全社活用の取り組みや私からのインプットなども通じて経営陣全体の解像度も上がってきており、最上位の「事業の価値」だけにフォーカスした高レイヤーな議論が行われる環境となっています。

## 運用のための取り組み

ここまで説明してきたような取り組みを全社のメンバーを巻き込んで効果的に実現するためには、各個人のモチベーションや理解度をうまく上げるための能動的な仕掛けが必要です。  
実際に Luup 社内で行われて割とワークしている、2つの取り組みについて紹介します。

### AI コミュニティ

Software 部内で AI 導入に意欲的・理解の深いメンバーを中心に **AI コミュニティ** というタスクフォース的なチームを組成し、定例会および Slack での質問・情報集積用の専用チャンネルを開設しました。

![ai-community-cycle](/images/etc-okada-20251225/ai-community-cycle.png)

定例会では直近の利用コスト、生産性向上の効果測定、直近話題のツールやモデルの情報共有、雑感などを話す技術者向けの議論が行われている一方、  
AI コミュニティでは使い方わからない時の相談、こんなニュース見た共有、こんなことできたという喜びの呟き、こんなことができたらいいなという緩い要望など、オープンな議論が交わせる場所になっており、Gemini Enterprise をはじめとする全社 AI リテラシー向上に一役買っています。

### 生成 AI 勉強会

「生成 AI を使えばこんなことができるのでは？」という着想を行動に移して、世の中の事例も調べながらあれこれ試してみる。そんな行動サイクルはエンジニアであれば自然に身についているかもしれませんが、技術に馴染みのない人にとっては中々簡単なことではありません。

そもそも生成 AI とはどういうもので、これ（全社員に配布されている Gemini）を使えばどんなことができるのかを実感と共に知ってもらうため、社内で不定期の勉強会を開催しています。

![ai-study-group](/images/etc-okada-20251225/ai-study-group.png)

こちらは今年8月に行われた勉強会の様子で、Web 上で動作する簡単なゲームアプリをお題に沿って要件定義から提出用サイトへのデプロイまで一通りやってもらうハンズオンを実施しました。  
オフラインとオンラインのハイブリッドで約40人もの参加者が来てくれましたが、0→1の想像以上の簡単さと、逆にそれを微修正する時の難しさについて実感し、楽しんでもらうことができました。

楽しんでもらえただけでなく、勉強会の実施後は、事業部側のメンバーから実現したい機能要件のプロトタイプを実際に動く画面で見せられるなどの場面が生まれており、クリエイティブの向上と AI への理解が進んだ良い会となりました。

## まとめ

この半年強、開発組織内外でさまざまな視点から AI 活用の推進をしてきましたが、遅ればせながら今までの仕事のあり方とは全く違った世界が広がってきています。  
現在では先進的な Web 業界の会社とも引けを取らないくらい高レベルな活用ができている状態になったと自負していますが、導入を始めてからの運用改善のスピードには目覚ましいものがありました。

優秀なメンバーが自らの業務を効率的・高品質に進めることを意識して実現できる組織においては、トップがあれこれ考えるのではなくまず（後戻りできるところまで）裁量と予算をいち早く用意して、走りながら効果検証をするのが大事だな…と強く実感しました。

もちろん課題はまだまだたくさんあるので、組織のあり方の定義も含めて更に AI ナイズドされた事業開発の仕組みを目指していきます。
